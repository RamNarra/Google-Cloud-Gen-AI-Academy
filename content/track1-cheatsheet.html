Three Agent Types in ADK
Pick the Right Type for the Job
LLM Agent
Language model is the brain. Handles reasoning, tool-calling, decision-making dynamically.
Use when: task requires reasoning and adapting to unpredictable input
Workflow Agent
Manages execution order of multiple agents. Three sub-types: Sequential, Parallel, Loop.
Use when: you know the order of operations and want to control the flow
Custom Agent
Full custom logic. You implement everything. For unique business logic that doesn't fit other patterns.
Use when: your use case is too specific for any standard workflow pattern
Workflow Agent Sub-Types
Sequential · Parallel · Loop — Know All Three
Sequential
Agents run one after another in order. Output of step N becomes input to step N+1.
Lab use: Researcher → Formatter
Parallel
All sub-agents start simultaneously. Independent tasks. Faster for multi-source research.
Use: search Wikipedia + DB + API at the same time
Loop
Repeats until condition met OR max iterations reached. Enables self-correction.
Example: coding agent loops until all 10 test cases pass (max 5 tries)
Cloud Run — Why You Need It
Local ≠ Production. Cloud Run Bridges the Gap.
Serverless — no infra to manage. Google handles OS, hardware, patching.
Scale to zero — when nobody uses it, it shuts down. Cost = ₹0 when idle.
Auto-scales up — you set a max. Traffic spikes handled automatically.
Low latency to Gemini — both on Google's internal network. Faster than calling from your laptop.
Built-in logging — every request, every tool call, every error is logged automatically.
Also good for MCP servers — deploy your own MCP tools here for Track 2.

output_key — stores agent output under a named key. Next agent reads it by key name instead of getting the full context. Keeps things efficient and cheap.
transfer control — root agent hands over completely to the sub-workflow. Compare to "agent as tool" where root stays in control and just calls the sub-agent for its output.

Project File Structure
Four Files — Know What Each Does
__init__.py
Python package entry point. Imports root_agent. ADK uses this to find your agent.
agent.py
Everything lives here — imports, logging, tool definitions, all agent definitions, root_agent.
requirements.txt
google-adk==1.14 (or latest), langchain, wikipedia. Create venv → pip install.
.env
PROJECT_ID, PROJECT_NUMBER, SERVICE_ACCOUNT, MODEL. Never commit to GitHub.
Full Deployment Sequence
Run These in Order — All on Google Cloud Shell
  • 01
    Go to console.cloud.google.com → Create project → Get Project ID (≠ Project Name)
  • 02
    Open Cloud Shell at shell.cloud.google.com → Authorise
  • 03
    Enable APIs: Cloud Run, Artifact Registry, Cloud Build, Vertex AI, Compute
    gcloud services enable run.googleapis.com artifactregistry.googleapis.com cloudbuild.googleapis.com aiplatform.googleapis.com compute.googleapis.com
  • 04
    mkdir zoo-guide-agent && cd zoo-guide-agent → open folder in Cloud Shell Editor
  • 05
    Create all four files: __init__.py, agent.py, requirements.txt, .env
  • 06
    Get env vars:
    export PROJECT_ID=$(gcloud config get-value project)
export PROJECT_NUMBER=$(gcloud projects describe $PROJECT_ID --format="value(projectNumber)")
  • 07
    Create service account (agent identity — least privilege):
    gcloud iam service-accounts create zoo-agent-sa
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="serviceAccount:zoo-agent-sa@$PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/aiplatform.user"
  • 08
    source .env → load env vars into terminal session
  • 09
    Create venv → activate → install:
    python3 -m venv venv && source venv/bin/activate && pip install -r requirements.txt
  • 10
    Deploy:
    adk deploy cloud run \
  --project $PROJECT_ID \
  --region asia-south1 \
  --service-name zoo-tour-guide \
  --with-ui \
  --service-account $SERVICE_ACCOUNT \
  ./
  • 11
    Type yes when prompted to create Artifact Registry Docker repo → wait 5–10 min → get URL → browse to it
  • 12
    Clean up when done: delete Cloud Run service → delete Artifact Registry repo → or delete whole project
What adk deploy cloud run Actually Does
Under the Hood — Four Automatic Steps
Packages your Python code into a Docker container image (you write no Dockerfile)
Pushes the image to Artifact Registry (private container repo inside your project)
Deploys the container to Cloud Run with your config — service name, region, service account
Spins up a web UI (because of --with-ui) — you get a chat interface at the returned URL
Security — Tuya Kept Repeating These
Non-Negotiable Practices
RULE 1
Least privilege on service accounts. Grant only roles/aiplatform.user. Nothing more. The agent does not need your full user permissions.
RULE 2
Never commit .env to GitHub. Not even to a private repo. Add .env to .gitignore before anything else.
RULE 3
Don't share Cloud Run URL publicly if unauthenticated. Anyone with the link can talk to your agent and burn your credits.
RULE 4
Add auth before premium features. Booking, payments, personal data — put a firewall/OAuth layer in front. Never expose these unauthenticated.
RULE 5
If commands fail — rerun them. Most setup commands are idempotent. Running twice won't break anything. Rerun before panicking.
