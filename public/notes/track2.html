02 The Evolution of AI — Why This Track Exists

Rahmen opens with a quick contextual arc — where AI has been and where we are now. Understanding this arc is important because it explains exactly why connecting agents to data is the unsolved challenge this track addresses.

Stage 1 — Rules-Based AI

The earliest form of AI was entirely deterministic. A developer explicitly coded every possible path, handled every edge case, and defined every output. The logic was clear but extremely brittle — anything outside the explicitly programmed paths would fail. The system was only as capable as the developer's ability to anticipate every scenario in advance.

Stage 2 — Traditional Machine Learning

Classification, regression, prediction — these models learned patterns from data rather than following hand-coded rules. More powerful, but with their own constraints: you needed large, clean, correctly labelled datasets, significant tuning effort, and the outputs were only as reliable as the training data.

Stage 3 — Generative AI (Large Language Models)

A wave of models trained on vast knowledge bases, capable of generating text, images, video, and other content from a simple prompt. Transformative, but with one fundamental limitation: the knowledge is frozen at training time. These models cannot access your internal data, cannot know what happened after their training cutoff date, and cannot act on the world — they can only respond to prompts.

Stage 4 — Autonomous Agents (Where We Are Now)

Agents go beyond responding. They understand the user's intent, reason about what steps to take, plan a sequence of actions, execute those actions, receive feedback, and continue — sometimes handing off to other agents. They are goal-directed. Rahmen puts it directly:

"The utility of these agents is going to come only if they are going to be acting on your data — or in other words working with your own data. You don't want the limitations of traditional generative AI models which are cut off in terms of their knowledge. It's definitely not going to be your internal data."

This is the motivation for Track 2: agents are only as useful as the data they can access. And connecting agents to real-world data in a reliable, maintainable, scalable way is the unsolved problem this session teaches you to solve.

03 The Core Problem — Why Plain APIs Are Not Enough

Before explaining MCP, Rahmen spends time establishing why the obvious solution — just let the agent call your existing APIs — is insufficient. This is worth understanding clearly, because it explains what problem MCP is actually solving.

The traditional approach: you put an API in front of your database or service, and any client application (including an AI agent) calls that API to get data. We have been doing this for years. So what is the problem?

  • APIs change. New versions are released. Breaking changes happen. Client applications that were integrated against an old version suddenly fail.
  • Boilerplate accumulates. Every new data source requires you to write connection code, handle authentication, manage retries on failure, implement error handling, and maintain all of it. Each new integration multiplies this burden.
  • There is no standard. Every API speaks a slightly different language. Every integration is a bespoke piece of work. There is no common way for an AI agent to discover what a service is capable of, what tools it exposes, or how to call them.
  • Discovery is missing. When a new capability is added to a backend service, the agent has no automatic way to learn about it. The developer must manually update the integration code.
What Was Actually Needed

A standardised protocol that defines: how a service describes its capabilities; how an AI client discovers those capabilities automatically; and how changes to backend services propagate to the client without requiring manual code updates. That is what MCP was built to provide.

04 What Is MCP?

MCP stands for Model Context Protocol. It was introduced by Anthropic and has since been adopted by Google as a standard across many of its services.

The simplest way to understand it: MCP is to AI agents what HTTP is to web browsers. HTTP standardised how browsers communicate with web servers. MCP standardises how AI agents communicate with tools and data sources.

What MCP Defines

The protocol specifies two sides of the connection:

AI Application / Agent (MCP Client) ⟷ MCP Protocol ⟷ Data Sources / Tools (MCP Server)
Your ADK agent · Gemini CLI · Cloud Code · Any AI tool                Databases · APIs · Google Cloud services · External tools

On the server side (MCP Server): a service describes its capabilities — what tools it has, what each tool does, what inputs it takes, what it returns. This metadata is machine-readable.

On the client side (MCP Client): the AI application reads the metadata, discovers what the server can do, and knows how to invoke the tools without the developer writing bespoke integration code for each service.

What This Means in Practice

Rahmen gives a concrete example. If a user asks an agent "what are the directions from here to the nearest coffee shop?" — the agent, using MCP, discovers that it has access to a Maps MCP server that has a tool capable of calculating distances and returning directions. It invokes that tool automatically. The developer did not write any routing logic. The agent figured out which tool to use from the metadata, called it, and returned the result.

Why Google Adopted It

Google has made MCP the standard for connecting its own services to AI agents. Any investment you make in learning MCP is not Google-specific — because MCP is an open, cross-industry standard, the same protocol works with any MCP-compatible AI tool. Write once, integrate everywhere.

The Two Sides of the Ecosystem

SideWhat it isExamples
MCP ClientThe AI application that wants to use toolsYour ADK agent, Gemini CLI, Claude Code, VS Code Copilot, any chat interface
MCP ServerThe service that exposes tools via the protocolA database, a Google Cloud service, a third-party API, your own custom backend

05 Two Options Google Provides

Google gives you two distinct ways to connect your agents to data via MCP. Rahmen is careful to present both without pushing you towards one — because both serve different needs. Understanding the difference is important before choosing which to use for your project.

OptionWhat it isBest for
MCP Toolbox for DatabasesOpen-source binary. You download and run it yourself — locally or on any server.Your own databases, custom data sources, local development, maximum flexibility
Google Managed MCP ServersFully managed MCP servers hosted by Google. You get an endpoint — no setup, no hosting.Google Cloud services (BigQuery, Maps, GKE, GCE). Production scale, zero infrastructure management.

Rahmen frames this clearly: the Toolbox is the flexible, open option. The Managed Servers are the convenient, scalable option. For a project that uses Google Cloud services and wants to get to production fast, the Managed Servers are the right choice. For a project that needs to connect to a non-Google database, or one running locally, the Toolbox is the right choice. You can also use both together.

06 Option 1 — MCP Toolbox for Databases

The MCP Toolbox for Databases is an open-source project. Its entire purpose is to eliminate the boilerplate code that you would otherwise need to write every time you want to connect an application to a database. It handles connection pooling, dropped connections, retries, observability, and query execution — and exposes all of this to your AI agent through a standard MCP interface.

Key Properties

  • Open source — the full source code is publicly available. You can inspect it, contribute to it, and modify it.
  • Available as a self-contained binary — you download a single executable file for your operating system (Linux, macOS, Windows). There are no dependencies to install. It runs anywhere.
  • Deploy anywhere — run it on your local laptop for development, or deploy it to Cloud Run (or any other server) for production. The configuration is identical either way.
  • Not locked to Google databases — while it supports Cloud SQL, BigQuery, AlloyDB, and Spanner, it also supports open-source databases (MySQL, PostgreSQL, SQLite) and commercial databases through community contributions.

Supported Databases

CategoryDatabases Supported
Google CloudBigQuery, Cloud SQL, AlloyDB, Spanner
Open SourceMySQL, PostgreSQL, SQLite
CommercialVarious — see official documentation for full list
LocalAny supported database running on your own machine

How the Toolbox Works — The Architecture

ADK Agent → MCP Toolbox Server
running on port 5000 → Your Database
BigQuery / Cloud SQL / MySQL / etc.

The Toolbox sits between your agent and your database. Your agent does not need to know anything about database connection strings, SQL syntax, authentication, or connection management. It simply calls the tool that the Toolbox exposes, and the Toolbox handles everything else.

Configuration — The tools.yaml File

The entire configuration of the Toolbox happens in one single file: tools.yaml. This is the beauty Rahmen keeps returning to — all the complexity of database connectivity is reduced to one readable configuration file. Here is what it looks like:

# tools.yaml — the ONLY file you need to configure the Toolbox

sources:
  mybq_source:          # any name you choose
    kind: bigquery       # or: cloud-sql, alloydb, postgresql, mysql, etc.
    project: your-project-id   # replace with your actual project ID

toolsets:
  my_bq_toolset:         # any name you choose
    tools:
      - search_release_notes_bq

tools:
  search_release_notes_bq:
    kind: bigquery-sql
    source: mybq_source
    statement: |
      SELECT product_name, description, published_at
      FROM `bigquery-public-data.google_cloud_release_notes.release_notes`
      WHERE DATE(published_at) >= DATE_SUB(CURRENT_DATE(), INTERVAL 7 DAY)
      ORDER BY published_at DESC

That is the entire configuration. You define your data source, define what tools exist, and write the SQL query that each tool executes. When your agent calls the tool named search_release_notes_bq, the Toolbox runs that SQL against your BigQuery dataset and returns the results. No connection code. No authentication handling. No retry logic. It is all abstracted away.

Key Insight — Replace the Query, Change the Use Case

The tools.yaml file shown above uses a Google Cloud public dataset as an example. In your own project, you replace the SQL statement with a query against your own data. Your agent then answers questions about your data instead of Google's release notes. The architecture is identical. Only the configuration changes.

07 Code Lab 1 — BigQuery + MCP Toolbox + ADK Agent

The first code lab builds a complete pipeline: a BigQuery public dataset, exposed through the MCP Toolbox, consumed by an ADK agent. The agent can then answer natural language questions about Google Cloud release notes.

What the Lab Builds

ADK Agent ↓ asks question in natural language
MCP Toolbox Server (port 5000) ↓ executes SQL query
BigQuery — google_cloud_release_notes

The user asks: "Give me the Google Cloud release notes from the last 7 days." The agent figures out that it has a tool (search_release_notes_bq) that retrieves this information. It calls the tool. The Toolbox executes the SQL. The result comes back to the agent. The agent formats and presents the answer.

Step-by-Step Lab Walkthrough

1

Verify your project. Open Cloud Shell at shell.cloud.google.com. Run gcloud config list project to confirm you are working in the correct project. If not, run gcloud config set project YOUR_PROJECT_ID. This is the most common mistake — running commands against the wrong project.

2

Identify your dataset. The lab uses the BigQuery public dataset bigquery-public-data.google_cloud_release_notes.release_notes. This is already available to everyone. You can preview it by running a query in the BigQuery console. This will be the data your agent answers questions about.

3

Download the MCP Toolbox binary. Create a folder, navigate into it, and download the binary for your OS. Since Cloud Shell is a Linux VM, use the Linux version: 

mkdir mcp-toolbox && cd mcp-toolbox
curl -O https://storage.googleapis.com/mcp-toolbox-releases/v0.x.x/toolbox_linux_amd64
chmod +x toolbox_linux_amd64
Check the releases page for the latest version number. Rahmen notes you can use the version specified in the lab or download the latest — either will work.

4

Create tools.yaml. Inside the same folder, create the configuration file. Define your BigQuery source, your toolset, and your SQL query tool. Replace your-project-id with your actual project ID. The structure is exactly as shown in Section 06 above.

5

Start the Toolbox server. Run the binary with your configuration file: 

./toolbox_linux_amd64 --tools-file tools.yaml
The output should say: initialised 1 source, 1 toolset with 1 tool, ready to serve. It listens on port 5000 by default. If port 5000 is in use, add --port 5001 (or any available port).

6

Test via the web UI. Run the toolbox with the --ui flag to get a visual testing interface: 

./toolbox_linux_amd64 --tools-file tools.yaml --ui
In Cloud Shell, click the Web Preview button, change the port to 5000, and click Preview. You should see: a hello-world confirmation, and at /api/toolset, the metadata of your exposed tools. You can also run the tool directly from the UI and see the results before writing a single line of agent code. Always test the Toolbox before integrating it with your agent.

7

Create the ADK agent. In a new terminal tab (keep the Toolbox running in the first), create a Python environment, activate it, and install dependencies: 

python3 -m venv venv && source venv/bin/activate
pip install google-adk toolbox-core
Note the additional package: toolbox-core. This is the MCP Toolbox Python client library that lets your ADK agent connect to the Toolbox server.

8

Write agent.py. The agent is straightforward — the key change from a standard ADK agent is how the tools are loaded: 

from google.adk.agents import Agent
from toolbox_core import ToolboxClient

# Connect to the Toolbox server running on port 5000
toolbox = ToolboxClient("http://localhost:5000")

# Load the tool set you defined in tools.yaml
tools = toolbox.load_toolset("my_bq_toolset")

# Create the ADK agent with those tools
root_agent = Agent(
    name="release_notes_agent",
    model="gemini-2.5-flash",
    instruction="You help users find and understand Google Cloud release notes.",
    tools=tools
)
That is the complete integration. Three lines connect your agent to your database.

9

Run and test. With the Toolbox server still running in the other terminal: 

adk run my_agent_folder/
Type: "Give me the Google Cloud release notes from the last 7 days." The agent calls the tool, the Toolbox executes the SQL, and the results come back. Try variations — last 2 days, last 3 days, specific products.

What You Just Proved

With no direct database connection code in your agent, no SQL in your Python, no authentication handling, and no retry logic — your agent is answering questions about live data from BigQuery. The entire database layer is abstracted behind one YAML file and one Python import.

08 Option 2 — Google Managed MCP Servers

In December 2024, Google announced a set of fully managed MCP servers — a different approach to connecting agents to Google Cloud services. These are not tools you download or run. They are endpoints hosted by Google that you simply enable in your project and configure in your agent.

First Wave of Managed MCP Servers

MCP ServerWhat it exposes
BigQuery MCP ServerQuery your BigQuery datasets, discover table schemas, execute SQL — all without writing SQL yourself
Google Maps MCP ServerSearch places, calculate routes, get directions, look up location data
GKE MCP ServerManage Google Kubernetes Engine clusters and workloads
GCE MCP ServerManage Google Compute Engine virtual machines

Rahmen notes that more managed servers will be added over time. The four listed above were the first wave at the time of this session.

Key Differences from the MCP Toolbox

AspectMCP ToolboxGoogle Managed MCP Servers
HostingYou run it. Local laptop or your own server.Google runs it. You get an endpoint URL.
SetupDownload binary, write tools.yaml, start server.Enable in your project via gcloud, configure API key if needed.
Databases supportedAny — Google Cloud, open source, commercial, localGoogle Cloud services only (in the first release)
ScalabilityDepends on where you host itGoogle-grade scalability, best-in-class reliability
IAM integrationBasicFull Google Cloud IAM — control exactly which accounts can invoke which servers
CostYou pay for the infrastructure you useManaged by Google — see documentation for pricing
Cross-product queriesRequires combining multiple Toolbox configsMultiple managed servers combine seamlessly in a single agent

How to Enable Managed MCP Servers

Managed servers are enabled via the gcloud CLI — they are integrated directly into the Google Cloud command-line tooling:

# List which MCP servers are currently enabled in your project
gcloud beta services mcp list

# Enable the BigQuery MCP server
gcloud beta services mcp enable bigquery

# Enable the Maps MCP server (also creates an API key automatically)
gcloud beta services mcp enable maps-tools

Once enabled, each server has a stable endpoint URL. Your ADK agent connects to that URL using the same MCP client pattern — no custom code, just a URL and authentication.

The Power of Combining Managed Servers

The real strength of Google Managed MCP Servers is cross-product queries. A single user prompt can trigger calls to two or more MCP servers simultaneously. The agent combines the results intelligently — without you writing any client code for either service. You will see this in action in Code Lab 2.

09 Code Lab 2 — Bakery Intelligence Agent (BigQuery + Maps)

The second code lab demonstrates the full power of combining managed MCP servers. The scenario is a fictitious bakery business that wants to expand. The agent has access to two data sources — a BigQuery dataset containing the business's own internal data (sales history, foot traffic, prices, demographics) and the Google Maps MCP server. A single natural language prompt can trigger the agent to query both sources, combine the results, and produce a rich, actionable answer.

The Bakery Dataset

A custom BigQuery dataset is set up by the lab's setup scripts. It contains four tables:

TableWhat it contains
bakery_pricesProduct names, prices, and which region they are sold in
sales_historyHistorical sales data by product, region, and date
foot_trafficHourly foot traffic measurements for current locations (e.g. 9–10am, 11am–12pm)
demographicsDemographic profiles of the areas around current and potential locations

Example Prompts the Agent Can Answer

These are taken directly from the code lab — and they are worth reading carefully, because they show how much reasoning and cross-system integration is happening behind a single sentence:

"I want to open a bakery in Los Angeles. Find the zip code with the highest morning foot traffic score."
"Can you search for existing bakeries in that area to see if the market is saturated? Give me the results with map links."
"I want a revenue projection for December 25. Look at my sales history for similar products and do a forecast."
"I want to open a new bakery. Find a neighbourhood with early morning activity, based on foot traffic data, and find the zip code with the highest score."

For the first prompt alone, the agent needs to: query the foot_traffic table in BigQuery, aggregate by zip code, identify the highest score, and return the result. For the second, it then calls the Maps MCP server to search for competing bakeries nearby and returns URLs to each location. All of this happens from a single sentence.

How the Agent Is Wired Up

The tools.py file in the lab defines two tool sets — one for each MCP server:

from google.adk.tools.mcp import MCPToolset

# Google Maps MCP Server — fully managed, uses API key for auth
def get_maps_tools():
    return MCPToolset(
        connection_params={
            "url": "https://maps.googleapis.com/mcp/v1",
            "api_key": os.environ["MAPS_API_KEY"]
        }
    )

# BigQuery MCP Server — fully managed, uses Google auth (ADC)
def get_bigquery_tools():
    return MCPToolset(
        connection_params={
            "url": "https://bigquery.googleapis.com/mcp/v1"
        }
    )

The agent.py file then combines both tool sets into a single agent:

from tools import get_maps_tools, get_bigquery_tools

root_agent = Agent(
    name="bakery_intelligence_agent",
    model="gemini-2.5-flash",
    instruction="""
    Help the user make data-driven business decisions.
    For demographic, pricing, and sales data — use the BigQuery toolset only.
    For real-world location analysis, competition, and routes — use the Maps toolset.
    Always include hyperlinks to locations when relevant.
    """,
    tools=[*get_maps_tools(), *get_bigquery_tools()]
)

What the Agent Does Behind the Scenes (Per Rahmen's Demo)

When the user asks about opening a bakery in Los Angeles, here is the sequence of tool calls the agent makes — automatically, without any hardcoded logic:

1

Calls BigQuery MCP to discover the available datasets and table schemas in the project.

2

Constructs a SQL query to aggregate foot traffic by zip code — without you writing the SQL. The agent writes the query itself based on its understanding of the task.

3

Executes the query and receives the zip code with the highest morning foot traffic score.

4

Calls the Maps MCP server to search for existing bakeries in that zip code.

5

Combines all results and presents a complete answer with location URLs, data-backed reasoning, and actionable recommendations.

The Critical Point

You did not write a single SQL query. You did not write a single line of Maps API code. You did not tell the agent which data source to use for which part of the question. The agent inferred all of that from the tool metadata and the user's natural language prompt. This is what MCP enables.

Setup Scripts — How the Lab Environment Is Created

The code lab uses a GitHub repository with pre-built setup scripts. Here is what they do:

1

setup_environment.sh — enables all required Google Cloud APIs, enables the BigQuery and Maps MCP servers in your project, and creates a Google Maps Platform API key. It also writes the API key into your agent's .env file automatically.

2

setup_bigquery.sh — creates a Google Cloud Storage bucket, uploads the CSV data files (prices, demographics, sales history, foot traffic), creates the BigQuery dataset, creates all four tables, and loads the CSV data into them.

3

After both scripts run, your BigQuery dataset is ready, your MCP servers are enabled, and your API keys are configured. You are ready to run the agent.
