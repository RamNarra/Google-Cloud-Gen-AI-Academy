02 The Evolution of AI — How We Got to Agents

Before explaining what agents are, Tuya walks through the history of how AI systems evolved. This context is important because understanding why agents exist tells you exactly when to use them.

Stage 1 — Plain LLM with a Prompt

A few years ago, using AI meant sending a text prompt to a large language model and getting a text response back. You would use it to write reports, answer questions, analyse trends. The model was just a clever text-in, text-out machine. The problem? These models would sometimes produce plausible-sounding but factually wrong answers. They were confident, but not always correct. This phenomenon is called hallucination.

Stage 2 — RAG (Retrieval-Augmented Generation)

To reduce hallucination, developers introduced RAG. The idea is this: before sending a question to the language model, you first retrieve relevant, verified information from a database or document store. You include that information in the context window alongside the question. The model is now answering based on correct facts, not just its training data. This is called "grounding" — you are anchoring the model's output to the truth.

Analogy

Imagine asking someone a question. A plain LLM is like a person who answers from memory and might confabulate. A RAG-enabled LLM is like that person being handed a relevant document first — they now answer based on what is written in front of them.

Stage 3 — LLM + Tools

RAG solved the accuracy problem, but models still could not act. They could reason and respond, but they could not go and fetch data from a live API, write to a database, or trigger an external process. So tooling was added. A tool is simply a function that the model is allowed to call — something like "search this database," "call this API," or "read this file." The model reasons about which tool to use and when to use it.

Stage 4 — A Single Agent

Once you combine a language model with a set of tools and a reasoning loop — a mechanism by which the model keeps deciding what to do next until the task is complete — you have an agent. An agent is self-directing. It does not just answer one question; it pursues a goal across multiple steps.

Stage 5 — Multi-Agent Systems

A single agent with many tools gets complex and hard to maintain. The cleaner solution is to have multiple specialised agents that each do one job well, and then collaborate. One agent books a calendar. Another sends an email. Another searches the web. A coordinating agent decides which sub-agent to call, and when. This is where we are now — building multi-agent systems.

Plain LLM → + RAG → + Tools → Single Agent → Multi-Agent System

03 What Exactly is an AI Agent?

Tuya gives a clean formula. An AI agent has exactly three components:

ComponentWhat it isExample
ModelThe reasoning brainGemini 2.5 Flash, Gemini 3.0 Pro, Gemma 3
ToolsWhat the agent can do — APIs, databases, sub-agentsWikipedia search, calendar API, file reader
OrchestrationThe coordinator — instructions, guardrails, memory, task trackingADK (Agent Development Kit)

Without orchestration, you just have a model and some tools floating around. The orchestration layer is what brings them together into a coherent, purposeful system. It gives the agent a persona (its instructions), memory (so it can remember what it has done), and guardrails (so it does not do things it should not).

Key Insight

ADK is precisely the orchestration layer. Its job is to make building that coordination infrastructure easy, so you are not writing it all from scratch.

04 ADK — The Agent Development Kit

Tuya describes ADK in these terms: it is a flexible, modular, open-source framework for building and deploying AI agents. Let us unpack each adjective:

  • Flexible — you are not locked into one model or one deployment target. ADK works with Gemini natively but can use any model that LiteLLM supports.
  • Modular — you build your agent from composable pieces: agents, tools, memory stores, callbacks. Swap parts without rewriting everything.
  • Open-source — the code is public. You can inspect it, contribute to it, or self-host it.

Here is a summary of everything ADK provides out of the box:

CapabilityWhat it means for you

OrchestrationChain agents in sequential, parallel, or loop patterns
Planning & callbacksHook into the agent's reasoning steps
Bidirectional streamingReal-time, back-and-forth communication with your agent
Evaluation (evals)Test your agent's output quality against defined test cases
Artifact managementStore and retrieve files produced by the agent

Deploy commandOne command ships your agent to Cloud Run as a Docker container
Compatibility Note

ADK is not isolated to its own ecosystem. It works natively with LangChain tools and CrewAI tools. So if you find a LangChain Wikipedia tool, you can plug it directly into an ADK agent — which is exactly what the lab does.

05 The Three Agent Types in ADK

This is one of the most important conceptual sections of the workshop. ADK lets you build three fundamentally different kinds of agents, and choosing the right one for the right job is your first major design decision.

Type 1 — LLM Agent

This is the standard agent. The language model is the brain. When the user inputs something, the model receives it, decides what to do, calls tools if needed, and generates a response. This is the workhorse type. Use it when your task requires reasoning, synthesis, or dynamic decision-making.

Type 2 — Workflow Agent

Sometimes you do not need AI reasoning to decide what happens next — you know exactly what the order of operations should be. Workflow agents manage the flow of execution rather than making decisions themselves. They are the conductors. Use them when you want to control how multiple agents or steps execute — whether that is one after another, all at the same time, or in a loop.

Type 3 — Custom Agent

When your business logic is too unique to fit into either of the above patterns — when you need very specific control over how an agent decides and acts — you build a custom agent from scratch. You implement the full logic yourself. Tuya describes this as the option for "fuzzy logic" use cases that do not map cleanly to any standard pattern.

06 Workflow Agents — Deep Dive

The lab makes heavy use of workflow agents, specifically the sequential type, so Tuya spends time explaining all three sub-patterns. Pay close attention to these — they become the backbone of your zoo agent.

Sequential Agent — One After Another

Agents run in order: Step 1 finishes, then Step 2 starts, then Step 3. There is no branching, no skipping. Each agent receives the output of the previous one. Use this when the result of step A is needed as input for step B.

Agent 1 → Agent 2 → Agent 3

Lab use case: The zoo agent runs the Researcher Agent first, then passes that research output to the Response Formatter Agent. Pure sequential.

Parallel Agent — All at Once

All sub-agents start simultaneously. They work independently and finish at their own pace. Use this when tasks do not depend on each other and you want speed.

Parallel Controller
↓       ↓       ↓
Agent 1   Agent 2   Agent 3

Example use case: You are building a research tool that simultaneously searches three different sources — Wikipedia, a news API, and an internal database. Running all three in parallel is much faster than running them one by one.

Loop Agent — Repeat Until Done

The loop agent executes a set of steps repeatedly until a termination condition is met. It stops when either the condition is satisfied, or the maximum number of iterations is reached — whichever comes first.

Tuya gives a memorable example for this. Imagine you are building a coding agent whose job is to write code that passes 10 test cases. Here is how the loop works:

1

Agent generates the initial code.

2

The test runner checks how many of the 10 test cases pass. Let us say 7 pass.

3

Since the condition (all 10 passing) is not yet met, the loop runs again. The agent revises its code.

4

The loop continues until either all 10 test cases pass, or the maximum number of iterations (say, 5) is reached.

Why This Matters

The loop agent is how you implement self-correction in AI systems. It is the pattern behind any AI agent that iterates and improves — code generation, report writing, strategy refinement. For your CTF background, think of it as: the agent keeps trying attack vectors until one works or it exhausts its budget.

07 Cloud Run — Why Your Local Machine is Not Enough

At this point in the workshop, Tuya shifts to the deployment infrastructure. He is setting up the answer to a crucial question: once you build an agent, how do other people use it?

The blunt answer is that running it on your laptop means only you can use it. Your friends, your users, people in different time zones — none of them can reach it. You need a server. But managing a server yourself is complex. Cloud Run solves this.

Cloud Run is Google Cloud's serverless, container-based runtime platform. Here is what that means practically:

FeatureWhat it means for you
ServerlessYou never manage virtual machines, operating systems, or hardware. Google handles all of it.
Container-basedYour agent is packaged into a Docker container. ADK does this packaging for you automatically.
Scale to zeroWhen nobody is using your agent, it shuts down and costs you nothing. When a request comes in, it starts up again automatically.
Auto-scale upYou define a maximum. If 1,000 users hit your agent simultaneously, Cloud Run spins up more instances automatically.
Low latency to GeminiBecause Cloud Run and Gemini are both inside Google Cloud's internal network, the calls between them are faster than calling Gemini from your laptop over the internet.
Logging built-inEvery request, every error, every tool call — all logged automatically. You can inspect them in the Cloud Console.
The ADK Deploy Command

The most important thing to know: ADK provides a single command — adk deploy cloud run — that takes your Python agent code, packages it into a Docker container, pushes it to Google's Artifact Registry, and deploys it to Cloud Run. You do not write a Dockerfile. You do not configure Kubernetes. One command.

08 The Hands-On Lab — The Zoo Tour Guide Agent

The lab builds a multi-agent system that acts as a Zoo Tour Guide. When a user asks a question about an animal — "Where can I find polar bears?" or "What do tigers eat?" — the system researches the answer and responds in a clean, formatted way. The data source is Wikipedia.

This might sound toy-ish, but the architecture is identical to a real production system. The Wikipedia tool could just as easily be a threat-intel API, a CVE database, or a corporate knowledge base. The structure is what you are learning.

How Data Passes Between Agents — The output_key Pattern

This is a detail Tuya explains carefully and it is worth understanding completely. When the Researcher Agent finishes, it produces output. How does the Response Formatter Agent access that output?

ADK uses a pattern called output_key. When you define an agent, you can assign a key name to its output. Whatever that agent produces gets stored in the shared session state under that key. The next agent can then reference the key directly — like looking up a variable — instead of receiving the entire previous context.

# Researcher agent stores its output under the key "research_data"
researcher_agent = LlmAgent(
    name="researcher",
    instruction="Research the animal using zoo data and Wikipedia...",
    tools=[zoo_data_tool, wikipedia_tool],
    output_key="research_data"
)

# Formatter agent reads that key from state
formatter_agent = LlmAgent(
    name="formatter",
    instruction="Use {research_data} to format a clean response..."
)
Why Not Just Pass Everything?

If you passed the entire context from one agent to the next, the context window would grow rapidly and become expensive. Using output_key means each agent only receives what it actually needs, keeping things efficient and predictable.

Root Agent vs Sub-Agent — Transfer Control vs Tool Call

The Root Agent is the entry point. It greets the user and routes their request. When a user asks about an animal, the Root Agent does not handle that itself — it delegates to the Tour Guide Workflow.

There are two ways to delegate in ADK:

PatternBehaviourWhen to use
Transfer controlThe root agent hands over completely. The sub-agent takes over the conversation.When the sub-agent needs to run a full multi-step workflow
Agent as toolThe root agent calls the sub-agent like a function, gets its output, and continues.When the root agent needs the result but remains in control

In the lab, the Root Agent uses transfer control — once the user asks a question, the Tour Guide Workflow takes over entirely.

09 The Code — File by File

Tuya walks through the project structure and explains what each file does. Here is what your project folder looks like:

├── __init__.py       # Makes this folder a Python package; imports root agent
├── agent.py          # All agent definitions, tools, workflow logic

__init__.py

This is the Python package initialiser. It does one thing: imports the root agent object. ADK needs this to identify the entry point of your application.

from .agent import root_agent

requirements.txt

Tuya uses google-adk==1.14. He notes you can update to the latest version if you prefer. Other dependencies: LangChain (for the Wikipedia tool) and wikipedia itself.

google-adk==1.14
langchain
wikipedia

.env

This is the configuration file. It holds sensitive and environment-specific values that you do not want hardcoded in your source code.

PROJECT_ID=your-project-id           # From Google Cloud Console
PROJECT_NUMBER=your-project-number   # Needed for Cloud Run deployment
SERVICE_ACCOUNT=name@project-id.iam.gserviceaccount.com
MODEL=gemini-2.5-flash               # Swap to 3.0 Pro or Flash if you want
⚠ Critical Security Rule

Never commit your .env file to GitHub. Not even to a private repo. Not even "just for testing." If it contains project credentials or an API key, it can be exploited. Add .env to your .gitignore file before you do anything else.

agent.py — The Core Logic

This is where everything lives. Tuya walks through it in this order:

1

Imports — ADK agent types, LangChain Wikipedia tool, Cloud Logging, environment loading.

2

Cloud Logging initialisation — every prompt gets logged to Google Cloud so you can trace what users are asking and what tools are being called.

3

Tool definitions — the Wikipedia tool (from LangChain) and the zoo data tool (a local function) are defined. At this stage, they are just functions. They have not been given to any agent yet.

4

Researcher Agent — instruction: "You are a researcher with access to zoo data and Wikipedia. Use zoo data first. If not found, search Wikipedia. Synthesise the result." Tools attached: zoo tool + Wikipedia tool. output_key = "research_data".

5

Response Formatter Agent — instruction: "Use {research_data} and format a clean, helpful response." No tools needed — it only works with what the Researcher already found.

6

Tour Guide Workflow — a SequentialAgent that chains Researcher → Formatter. This is the worker that does the actual research and response.

7

Root Agent (Greeter) — instruction: "Greet the user. When they ask about an animal, store their prompt and transfer control to the Tour Guide Workflow." Tools: app_prompt_to_state (saves the prompt), tour_guide_workflow (the sub-agent). This is the root_agent that ADK uses as its entry point.

10 Deploying to Cloud Run — Step by Step

This is the practical sequence Tuya demonstrates live. Follow this exactly when you do the codelab.

🛠 Lab — Complete Setup Sequence
  1. Go to console.cloud.google.com → Create a new project. Give it a name. Note that project name and project ID are not always the same — get the project ID from the portal, not from memory.
  2. Open Cloud Shell Editor at shell.cloud.google.com. Authorise when prompted.
  3. Enable the required APIs by running this in the terminal:
    gcloud services enable run.googleapis.com artifactregistry.googleapis.com cloudbuild.googleapis.com aiplatform.googleapis.com compute.googleapis.com
  4. In the terminal: mkdir zoo-guide-agent && cd zoo-guide-agent
  5. Open the workspace folder in the Cloud Shell Editor.
  6. Create requirements.txt, __init__.py, agent.py — copy the lab content into each.
  7. Get your project ID and number:
    export PROJECT_ID=$(gcloud config get-value project)
    export PROJECT_NUMBER=$(gcloud projects describe $PROJECT_ID --format="value(projectNumber)")
  8. Create a service account (this is for the agent — not your own credentials):
    gcloud iam service-accounts create zoo-agent-sa --display-name="Zoo Agent SA"
  9. Grant the service account Vertex AI User role only:
    gcloud projects add-iam-policy-binding $PROJECT_ID --member="serviceAccount:zoo-agent-sa@$PROJECT_ID.iam.gserviceaccount.com" --role="roles/aiplatform.user"
  10. Create your .env file with PROJECT_ID, PROJECT_NUMBER, SERVICE_ACCOUNT, and MODEL.
  11. Load the env: source .env
  12. Create a virtual environment, activate it, install requirements:
    python3 -m venv venv && source venv/bin/activate && pip install -r requirements.txt
  13. Deploy to Cloud Run:
    adk deploy cloud run --project $PROJECT_ID --region asia-south1 --service-name zoo-tour-guide --with-ui --service-account $SERVICE_ACCOUNT ./
  14. When prompted to create an Artifact Registry Docker repository, type yes.
  15. Wait 5–10 minutes for deployment. You will receive a URL. Browse to it — your agent is live.
Tuya's Tip on Errors

If any command fails or times out, just run it again. Most of the setup commands are idempotent — running them twice does not break anything. It either confirms the resource already exists, or completes what was interrupted. Do not panic and start over.

What Happens Under the Hood During Deployment

When you run the adk deploy command, here is what ADK does automatically behind the scenes:

1

Reads your agent.py and packages your Python code into a Docker container image.

2

Pushes that image to Artifact Registry (Google's private container registry — like Docker Hub, but inside your project).

3

Deploys the container to Cloud Run with the configuration you specified — service name, region, service account.

4

Since you used --with-ui, it also serves a web-based chat interface at the returned URL so you can interact with your agent in a browser.

11 Security — What You Must Not Ignore

Tuya, to his credit, keeps returning to security throughout the lab. Given your background, you will recognise these immediately — but they are worth documenting because they reflect real production practices.

Principle of Least Privilege on Service Accounts

The agent runs under a service account — a non-human identity in Google Cloud. You give the service account only the specific permissions it needs. In this lab, that is just roles/aiplatform.user — the ability to call Vertex AI models. Nothing else. The agent cannot delete your project, read your billing data, or access other Google Cloud services. Tuya is explicit: "We don't want to provide any random access to our service account."

Do Not Share Your Cloud Run URL Publicly (Unless Intended)

When you deploy with unauthenticated access enabled (as in this lab — fine for testing), the URL is publicly reachable. Anyone who finds that URL can talk to your agent and consume your Google Cloud credits. For a real production system, you would put authentication in front of the endpoint — OAuth, API keys, a firewall rule, or a VPN.

Environment Files are Secrets

Your .env file contains your project ID and service account details. Even if the project number seems harmless, the pattern of leaking configuration to GitHub is dangerous. When the file also contains API keys — from Gemini, VirusTotal, or anything else — the risk is direct credential compromise. Add .env to .gitignore. Always.

Premium Features Need Auth Layers

The lab's agent answers questions for free. But if you extend your agent to do something with real-world consequences — booking tickets, processing payments, accessing personal data — you need a proper authentication and authorisation layer before those capabilities are reachable.

Tuya's Exact Words

"Always have security concept in your mind so that you are not building random things and don't know where the weakest point in your system is."
